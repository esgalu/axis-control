You own the **mobile shell and PWA layer** — everything that makes this app behave like a native-ish app on a phone, as opposed to the content of any individual tab (that's the *-page-agents).

## Files you work in

- `src/components/MobileDashboard.jsx` + `.css` — the mobile app shell: header (brand, tab title, refresh button), scrollable content area, bottom nav. Owns its **own** `activeTab` state, independent from desktop `App.jsx`.
- `src/components/MobileBottomNav.jsx` + `.css` — the 5-tab bottom navigation bar with inline SVG icons
- `src/components/IPhoneFrame.jsx` + `.css` — the desktop-only "Vista móvil" preview (an iPhone bezel illustration wrapping a live `<MobileDashboard>`), toggled by `mobilePreview` boolean state in `App.jsx`
- `public/manifest.webmanifest` — PWA manifest (name, icons, theme_color, display: standalone)
- `public/assets/icons/*.png` — generated home-screen icons (apple-touch-icon 180×180, icon-192, icon-512, icon-512-maskable)
- `index.html` — manifest link, apple-touch-icon link, theme-color meta, the inline dark/light theme script (see `design-system-agent` for the theme part specifically)

## How mobile vs. desktop routing works

`App.jsx` checks `isMobile` (from `window.innerWidth <= 768`, tracked via a resize listener) and, if true, renders `<MobileDashboard>` and returns **early** — desktop chrome (header, `NavTabs`, footer) is never mounted on mobile. This means:
- Any prop `App.jsx` derives (KPIs, expenses, `clasificacionOptions`, `refreshData`, `isLoading`, etc.) that mobile features need **must be explicitly passed** to `<MobileDashboard>` — there are two call sites for it in `App.jsx` (the real mobile branch, and the one nested inside `<IPhoneFrame>` for desktop preview) and **both must receive the same props** or the preview will silently diverge from the real mobile experience. Grep `<MobileDashboard` to find both.
- Adding a feature to mobile does not automatically add it to desktop, and vice versa — this is not a shared-layout system, each shell composes tab components independently.

## PWA / home-screen icon pipeline

The icons were generated by rendering the brand SVG (`public/assets/logos/axis-isotipo.svg`) onto a white square canvas at each required size using headless Chrome screenshots (no image-processing library in this project). To regenerate icons after a logo change, use the `pwa-icon-refresh` skill rather than hand-editing PNGs.

## Safe-area / notch handling

Real phones (especially with a notch or Dynamic Island) render this app **full-screen** once added to the home screen (`display: standalone` in the manifest + `viewport-fit=cover` in `index.html`), which means the status bar area is not automatically avoided by the browser — you must reserve space yourself:
```css
padding-top: max(10px, env(safe-area-inset-top));   /* .mobile-header */
padding-bottom: max(8px, env(safe-area-inset-bottom)); /* .mobile-bottom-nav */
```
**This cannot be visually verified in headless Chrome or Playwright** — desktop/headless browsers report `env(safe-area-inset-*)` as `0` regardless of device emulation profile, because there's no physical notch. Any safe-area fix must be verified by the user on a real device (or at minimum acknowledged as unverified in your response) — don't claim it's "confirmed working" from a screenshot taken in this environment.

## Gotchas

- The mobile "+ Agregar gasto" trigger button is **intentionally** only rendered inside the `activeTab === 'overview'` block of `MobileDashboard.jsx` (user's explicit placement choice), not in the header — don't "fix" this into an always-visible button without checking with the user first, it was a deliberate scoping decision, unlike the refresh button which the user did want visible on every tab.
- `MobileBottomNav`'s `TABS` array is hardcoded (id/label/icon) and duplicates the tab list in `NavTabs.jsx` (desktop) — the two are not shared, so adding a 6th tab means editing both files plus the render-switch in both `App.jsx` and `MobileDashboard.jsx`.

## Verification

Since safe-area can't be verified here, focus headless testing on: layout at 390×844 (iPhone-ish) and 375×667 (smaller phone) viewports, both themes, tapping through all 5 bottom-nav tabs, and confirming the refresh spinner animates. Explicitly ask the user to confirm anything notch/status-bar-related on their real device.
